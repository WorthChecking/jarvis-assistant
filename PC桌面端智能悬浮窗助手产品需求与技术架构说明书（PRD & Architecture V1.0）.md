# PC桌面端智能悬浮窗助手产品需求与技术架构说明书（PRD & Architecture V1.0）

## 【需求基线】

* **目标操作系统**：Windows系统。
* **核心执行权限**：能够接管电脑进行操作；可以查找、访问和读取系统文件；严禁修改系统核心设置；严禁擅自删改系统中存储所有的内容。
* **架构选型**：大语言模型调用云端API；STT和TTS本地部署。
* **记忆机制**：引入本地向量数据库实现“长期记忆”。
* **UI视觉与交互**：动态波形（悬浮窗）。

---

## 版块一：整体架构与技术选型

### 1. 系统开发框架
* **表现层（UI端）**：**Tauri + Vue 3**
    * **选型逻辑**：作为桌面悬浮窗，常驻后台时不能占用过高系统资源。Tauri 使用系统原生 WebView 且底层基于 Rust，其内存和 CPU 开销远低于 Electron。采用 Vue 3 能够高效管理前端状态，流畅驱动高帧率的动态波形渲染。
* **控制层（本地服务端）**：**Python**
    * **选型逻辑**：Python 是 AI 模型推理（STT/TTS）、向量数据库操作以及 Windows 系统底层 API 调用的绝对标准环境。表现层与控制层之间通过本地 WebSocket 或轻量级 HTTP 接口实现毫秒级通信。

### 2. 核心 AI 引擎与模型选型
* **STT（本地语音识别）**：**SenseVoice-Small (基于 FunASR)**
    * **选型逻辑**：阿里通义实验室开源的 SenseVoice-Small 在中文及多语种混合识别上准确率极高，且推理极速（50ms级别），对显存要求极低（不足1GB即可流畅运行）。完美契合桌面端低延迟交互需求。
* **TTS（本地语音合成）**：**Bert-VITS2 或 ChatTTS**
    * **选型逻辑**：为了达到“像真人类一般”的拟人化发音与自然停顿，传统 TTS 无法满足需求。VITS 架构或新型的 ChatTTS 支持高质量克隆与自然韵律，推理时显存占用可控制在 1.5GB 左右。
* **LLM（云端大脑）**：**深度求索 DeepSeek**
    * **选型逻辑**：该助手不仅需要对话，更需要触发系统底层操作。这要求云端 API 必须具备极其精准的 **Function Calling（函数调用）** 能力，将用户的意图转化为可执行代码指令。
* **长期记忆（本地向量数据库）**：**ChromaDB**
    * **选型逻辑**：极致轻量的纯本地向量数据库，可直接作为 Python 库嵌入运行，无需配置 Docker 或额额外部署独立的数据库服务端，极度契合单机个人助手的定位。

## 版块二：本地系统接管与操作白名单控制策略

### 1. 进程权限控制 (Privilege Management)
* **降权运行**：应用前端（Tauri）与后端（Python）必须严格以**普通标准用户权限**（Non-Administrator）运行。从 Windows 操作系统底层直接剥夺该程序修改注册表核心键值、篡改系统目录（如 `C:\Windows`）或进行静默提权的能力。

### 2. 指令执行架构 (Function Calling)
严禁允许 LLM 直接生成并执行终端脚本（如 CMD / PowerShell / VBScript）。云端 LLM 仅承担“意图解析大脑”的角色，实际执行必须依赖预设的函数调用。
* **执行链路**：用户语音输入 STT 文字 -> 上传给 LLM -> LLM 判定需要操作电脑 -> LLM 返回指定格式的工具调用指令（JSON） -> 本地中控层解析并校验白名单 -> 触发本地预写好的 Python 封包函数 -> 执行完毕并将状态反馈给 LLM -> 生成 TTS 语音回复。

### 3. 操作白名单池 (Action Whitelist)
以下为系统允许 AI 调用的预设动作集合。任何超出此范围的 LLM 请求，本地拦截器将直接抛出鉴权失败，并让 AI 回复“权限不足”。
* **设备环境控制**：系统音量调节（增减/静音）、多媒体控制（播放/暂停/切歌）、当前前台窗口最小化/最大化、系统锁屏。
* **应用启停管理**：仅允许通过扫描“开始菜单”或“桌面快捷方式”来启动已知应用程序。禁止应用具有根据 PID 强制结束（Kill）非自身派生进程的权限，防范 AI 误关重要系统服务。
* **文件系统检索**：接入本地 `Everything` (Voidtools) 的 SDK 或 HTTP 服务。当用户要求查找文件时，调用 `Everything` 接口实现毫秒级全盘绝对路径检索。检索黑名单强制排除 `$Recycle.Bin`、`System Volume Information` 及包含系统凭据的敏感路径。
* **文件内容读取**：授权本地服务解析并读取常用工作文档（.txt, .md, .pdf, .docx, .csv）的纯文本内容，提取后作为上下文传给 LLM。

### 4. 危险操作阻断器 (Safety Interceptor)
* 在 Python 中控代码层，对标准库文件操作进行 AOP（面向切面编程）拦截。强制封死并重写与删除、修改强相关的内置函数接口（如屏蔽 `os.remove`, `shutil.rmtree`, `os.rename` 等底层调用），确保即使发生越权调用，在代码执行层面也会被直接丢弃。

## 版块三：长短期记忆机制设计

### 1. 记忆分层架构
系统记忆分为“短期会话缓存”与“长期向量持久化”双层结构：
* **短期记忆（Short-term Memory - RAM层）**：
    * **机制**：基于滑动窗口（Sliding Window）的对话历史队列。仅保留当前交互周期内的最近 $N$ 轮对话（建议设定为最近5-10轮）。
    * **作用**：处理代词指代，保证即时对话的极速连贯，无需每次都进行重度向量检索。当悬浮窗进入休眠或超过预定时间未交互，该队列自动触发“摘要压缩”并清空。
* **长期记忆（Long-term Memory - 数据库层）**：
    * **机制**：基于 ChromaDB 构建本地向量库。
    * **作用**：持久化存储用户的个人偏好、历史操作习惯及过往陈述的核心事实。在每次用户发起新指令时，通过语义相似度检索，将相关历史信息跨会话提取。

### 2. 嵌入模型选型 (Embedding Model)
* **选型**：使用 `bge-small-zh-v1.5` 或 `all-MiniLM-L6-v2`。此类模型推理极快，可完全由 Intel i7 CPU 承担计算，不占用 RTX 3060 的核心显存。

### 3. 记忆生命周期管理 (CRUD 机制)
* **记忆写入 (Write/Summarize)**：
    * 引入“记忆决策器”逻辑：在每一轮独立会话结束后，由 LLM 判断是否存在需要记忆的“新事实”。若存在，则提取核心陈述，向量化后存入 ChromaDB。
* **记忆检索 (Retrieve)**：
    * 采用 Top-K 检索机制。文本向量化后，在 ChromaDB 中检索相似度最高的 $K$ 条记忆（设定 $K=3$）。将这 $K$ 条记忆作为 `System Prompt` 的补充部分，发送给云端 LLM。
* **记忆冲突与更新 (Update/Decay)**：
    * 若检索到的长期记忆发生冲突，必须在存入新记忆时，通过元数据（Metadata）中的时间戳进行覆盖，或由 LLM 判断执行旧记录的软删除（Soft Delete）。

## 版块四：核心交互链路与并发显存调度逻辑

### 1. 24小时低功耗唤醒机制 (Wake Word Engine)
* **策略**：采用纯 CPU 计算的极轻量级唤醒引擎（如 PicoVoice Porcupine）。该引擎只负责监听特定的唤醒词，当且仅当检测到唤醒词时，才向主控进程发送激活信号，唤醒后续的 STT 推理链路。

### 2. 全双工核心交互流水线 (Interaction Pipeline)
* **T0 - 唤醒与录音**：检测到唤醒词，前端 UI 切换为“聆听”状态，麦克风开启音频分块截取。
* **T1 - 语音转文本 (STT) 【V1.2修订】**：本地 SenseVoice-Small 模型接收音频张量。依靠 FunASR 的极速推理能力，在语音停顿的瞬间（毫秒级）输出完整的高精度文本，大幅降低后续链路的起步延迟。
* **T2 - 记忆检索**：文本传入本地 ChromaDB，毫秒级提取 Top-3 历史相关记忆。
* **T3 - 云端推理与动作拦截**：组装 Prompt 并向云端 LLM 发送请求。如果识别到 JSON 格式的工具调用，立即阻断语音生成，优先交由本地 Python 执行操作。
* **T4 - 文本转语音 (TTS) 极速首包流**：非操作类文本一旦从 LLM 回传达到首个完整标点符号，立刻送入本地 TTS 模型进行推理。
* **T5 - 播放与波形同步**：音频流输出至系统声卡，并同步发送音频响度数据给 Tauri 前端，驱动 UI 波形变化。

### 3. 并发显存防溢出与调度策略 (VRAM Management)
* **常驻显存策略（低延迟优先）**：
    * STT（SenseVoice-Small）：划定显存上限约为 1GB（使用 ONNX 或 PyTorch 均可轻松压制）。
    * TTS（VITS 或同等量级）：划定显存上限 1.5GB。
    * 替换引擎后，常驻显存总占用从 3.5GB 下降至 2.5GB 左右，RTX 3060 6GB 的安全性得到极大提升。
* **资源熔断机制**：中控 Python 进程每秒轮询一次 `nvidia-smi` 获取当前显存余量。一旦检测到显存使用率超过 90%，触发降级策略。

## 版块五：UI状态流转与动态波形渲染机制

### 1. 悬浮窗基础形态与窗口控制
* **无边框与透明底（Frameless & Transparent）**：通过 Tauri 原生 API 配置，剥离传统 Windows 窗口的标题栏与边框。
* **事件穿透机制（Click-Through）**：
    * 在“待机”状态下，使悬浮窗区域对鼠标事件完全穿透。
    * 在“聆听”或“交互”状态下，恢复局部点击响应。
* **层级锁定**：调用操作系统的 `Always on Top`（总在最前）接口，确保助手随时可见。

### 2. 核心状态机设计 (State Machine)
UI 仅允许在以下四种严格定义的内部状态中流转，禁止出现模棱两可的中间态：
* **State 0：待机 (Standby)**
    * **视觉**：默认冷色调。收缩为极简的静态微小粒子核心，呼吸频率极低，主动释放所有前端 GPU 渲染资源。
* **State 1：聆听 (Listening)**
    * **视觉**：默认冷色调。粒子核心展开为动态散点。粒子的活跃度与扩展半径与麦克风输入的实时音频响度严格挂钩。
* **State 2：处理中 (Processing)**
    * **视觉**：色彩体系强制平滑切换为橘黄色。引入三维空间坐标（3D视角），中心保持高密度核心，其余粒子如同地球上各种各样的卫星，在不同高度和倾角的线路上作三维交错环绕运转，形成错落交织的视觉感受，表示大脑正在运算且不接收新的环境噪音。
* **State 3：播报与执行 (Action & Speaking)**
    * **视觉**：色彩体系保持为橘黄色。如果是操作，UI 闪烁警戒色提示系统接管；如果是播报，粒子形态恢复至与 State 1（聆听）完全一致的结构，散点振幅与 TTS 输出音频的实时频率/能量包络同步跳动，模拟拟人化的发音口型变化。

### 3. 动态波形渲染技术方案 (Rendering Engine)
* **核心技术栈**：采用 **WebGL (如 Three.js) 或包含 3D 矩阵投影的 Canvas 2D** 进行底层绘制，严禁使用 Vue 的双向绑定或 CSS Keyframes 去频繁操作 DOM。
* **数据驱动逻辑**：
    * 前端通过 Web Audio API 的 `AnalyserNode` 获取音频流。
    * 利用 `requestAnimationFrame` 以 60fps 的固定帧率将数据映射为粒子系统的坐标变化与轨道旋转。
    * 引入对象池（Object Pool）技术与状态间缓动插值（Easing），确保在 2D 散点与 3D 轨道状态切换时，粒子不发生销毁重绘和瞬间闪跳。
