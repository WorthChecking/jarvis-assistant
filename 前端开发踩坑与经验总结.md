# JARVIS 前端开发踩坑与经验总结

> 基于 Three.js + Vue 3 + Tauri 的 PC 桌面端智能悬浮窗助手粒子渲染引擎开发经验

---

## 一、时间管理：抛弃 Three.js Clock

### 问题

冷启动时粒子运动忽快忽慢，但 HMR 热更新后恢复正常。

### 根因

`THREE.Clock` 的 `getDelta()` 和 `getElapsedTime()` 存在内部耦合：

```js
// ❌ 每帧 getDelta() 被隐式调用两次
const dt = this.clock.getDelta()         // 调用 1：返回真实 dt，更新 oldTime
const time = this.clock.getElapsedTime() // 调用 2：内部再次调用 getDelta()，oldTime 又被更新
```

第二次 `getDelta()` 消耗了 ~0.1ms 的时间差，导致下一帧 `dt` 测量起点偏移。冷启动帧率不稳定时偏移被放大，造成 lerp 速度波动。

### 修复

用 `performance.now()` 自管时间，`dt` 和 `time` 各算各的：

```js
private lastFrameTime = 0
private animTime = 0

// 首帧只记录时间戳
if (this.lastFrameTime === 0) {
  this.lastFrameTime = now
  return
}

const rawDt = (now - this.lastFrameTime) / 1000
this.lastFrameTime = now
if (rawDt <= 0.001 || rawDt > 0.1) return  // 拒绝异常帧

const dt = rawDt
this.animTime += dt
const time = this.animTime
```

### 经验

**永远不要同时调用 `Clock.getDelta()` 和 `Clock.getElapsedTime()`**。如果必须用 Clock，选一个用。更好的做法是完全抛弃 Clock，自管时间。

## 二、插值公式：`dt * k` 不是帧率无关的

### 问题

冷启动时粒子运动速度波动（忽快忽慢），HMR 后正常。

### 根因

```js
// ❌ dt 波动 → lerpSpeed 波动 → 移动速度波动
const lerpSpeed = dt * 5.0
posArr[i3] += (tx - posArr[i3]) * lerpSpeed
```

`dt * k` 形式的 lerp **不是帧率无关的**。当 dt 从 16ms 波动到 20ms（帧率 60→50），lerpSpeed 从 0.08 变为 0.10，速度差异 25%。

### 修复

帧率无关公式 `1 - e^(-k * dt)`：

```js
// ✅ 无论 dt 如何波动，收敛速度恒定
const posLerpK = 1 - Math.exp(-5.0 * dt)
posArr[i3] += (tx - posArr[i3]) * posLerpK
```

### 数学证明

指数衰减 `pos += (target - pos) * (1 - e^{-k \cdot dt})` 的解为：

$$x(t) = target \cdot (1 - e^{-k \cdot t})$$

收敛速度仅由 `k` 决定，与 `dt` 无关。

### 经验

**任何涉及位置/颜色/尺寸插值的场景，必须用 `1 - exp(-k * dt)` 替代 `dt * k`**。后者只在帧率绝对稳定时（如独立 HTML 文件）可用，在 Tauri/Electron 等壳体中帧率必然波动。

## 三、lerp 追踪正弦目标：振幅衰减与相位滞后

### 问题

lerp 追踪正弦目标时，粒子振幅只有目标的 55%，且运动不均匀。

### 根因

指数 lerp 对正弦目标的稳态响应为：

$$x(t) = \frac{k}{\sqrt{k^2 + \omega^2}} \cdot \sin(\omega t - \arctan\frac{\omega}{k})$$

当 `k=5, ω=2π×1.2≈7.54` 时，振幅衰减到 `5/√(25+56.9) ≈ 0.55`（仅 55%），相位滞后 `arctan(7.54/5) ≈ 56°`。

### 两种修复方案

**方案 A：直接赋值 + 状态过渡（零滞后）—— ❌ 已验证不可用**

```js
// 状态切换时快照位置
this.prevPositions = new Float32Array(this.positions)
this.transitionAlpha = 0.0

// 运行时直接赋值
posArr[i3] = tx  // 零滞后

// 过渡期间混合
const a = this.transitionAlpha
posArr[i3] = prevArr[i3] * (1-a) + tx * a
```

直接赋值跳过中间位置，与 fade plane opacity=0.15 的拖尾机制冲突：旧帧粒子在 85% 亮度残留，新帧粒子已跳到新位置 → 残影闪烁。将 fade opacity 提到 1.0 可消除残影，但动态切 0.15↔1.0 本身又会引入闪烁（见第十四条）。**此方案在当前渲染管线下不可用**。

**方案 B：提高 lerp k 值（牺牲平滑度换精度）—— ❌ 已验证不可用**

```js
// State 1: k=40, State 3: k=60
posLerpK = 1 - Math.exp(-40.0 * dt)
```

高 k 使粒子紧密追踪目标，波峰处速度接近零（过冲消除）。但高 k 导致每帧位移 = 目标位移 ≈ 2.5 units（~23px），远超粒子尺寸（~4px）。在 fade 0.15 下，旧帧粒子在 85% 亮度残留，与新帧粒子相距 23px → **离散残影点**。与直接赋值（方案 A）的残影问题本质相同：fade plane 的拖尾机制要求帧间位移远小于粒子尺寸。

**方案 C：速度预测前馈（k=5 不变，目标加入导数前馈）—— ✅ 最终采用**

```js
// k=5 不变，在目标位置上叠加 τ × 目标速度
const wavePhase = time * waveFreq - phaseOffset
const breathe = (Math.sin(wavePhase) + 1.0) * 0.5
const dynamicRadius = (baseRadius + 4.0) + (breathe * maxExpansion)
// 目标速度（dynamicRadius 的导数）
const radiusVel = Math.cos(wavePhase) * 0.5 * waveFreq * maxExpansion
// 预测目标 = 当前目标 + τ × 速度
const predictedRadius = dynamicRadius + radiusVel * tau
// State 1: τ=0.08, State 3: τ=0.05
tx = predictedRadius * cos(theta) * sin(phi)
// ...
posArr[i3] += (tx - posArr[i3]) * posLerpK  // lerp k=5 不变
```

### 速度预测的数学原理

lerp 对正弦目标 `T(t)=A·sin(ωt)` 的稳态响应有相位滞后 `φ=arctan(ω/k)`。若将目标替换为预测目标 `P(t)=T(t)+τ·T'(t)=A·(sin(ωt)+τω·cos(ωt))`，则：

- 预测目标的相位超前：`arctan(τω)`
- 残留相位滞后：`arctan(ω/k) - arctan(τω)`
- 当 `τ=1/k` 时，残留滞后 = 0（完美跟踪），但位移 = 目标位移（同方案 A/B 的残影问题）
- 当 `τ<1/k` 时，部分抵消滞后，位移增加可控

| τ | State 1 残留滞后 | State 1 波峰速度 | State 1 中心位移 | State 1 波峰位移 |
|---|---|---|---|---|
| 0（无预测） | 56° | 69.1 (基准) | 12.9px (基准) | 10.7px (基准) |
| 0.05 | 35° | 58.0 (-16%) | 13.9px (+8%) | 8.5px (-21%) |
| **0.08** | **25°** | **41.1 (-41%)** | **15.1px (+17%)** | **6.4px (-40%)** |
| 0.10 | 19° | 34.0 (-51%) | 16.2px (+26%) | 5.3px (-50%) |
| 0.15 | 7.5° | 16.4 (-76%) | 19.5px (+51%) | 2.5px (-77%) |
| 0.20=1/k | 0° | 0 (-100%) | 23.4px (+81%) | 0px (-100%) |

τ=0.12（State 1）和 τ=0.08（State 3）选在**波峰速度降低 70-80%、残留滞后 14-24°** 的平衡点。

**τ 值演进**：
- 初版 τ=0.08/0.05：波峰速度降低 40-25%，解决"中心平滑、外围卡顿"，但用户仍报告波峰附近"加速冲向最外圈"和"突然反向"感
- 最终 τ=0.12/0.08：波峰速度降低 80-70%，进一步减弱波峰加速感。lerp 振幅增加约 15%（vs τ=0.08/0.05），在 fade=0.3 下残影可控

**波峰加速感的根因**（稳态分析，State 1, τ=0.08）：

lerp 粒子在目标波峰（ωt=90°）处仍有 1.035A 的外冲速度（目标速度已为 0），因为 lerp 粒子滞后 25°。粒子以高速冲向已停止的目标 → "即将到最外圈时加速"。

过波峰后（ωt=115°→130°），lerp 粒子从 0 速度变到 -0.615A 只用 0.035s → "刚从最外圈往回走时加速"。

增大 τ 到 0.12 后，残留滞后 14°，波峰速度降到 0.69A（-33%），加速感减弱。

### 波峰过冲的深层分析

粒子在目标峰值处的速度是关键指标。目标 `T(t)=A·sin(ωt)` 在峰值处速度为 0，但 lerp 粒子的稳态速度为：

$$x'(t) = A \cdot \frac{k\omega}{\sqrt{k^2+\omega^2}} \cdot \cos(\omega t - \varphi)$$

在目标峰值（ωt=π/2）处，粒子速度为：

$$x' = A \cdot \frac{k\omega^2}{k^2+\omega^2}$$

| k | ω=7.54 (State 1) 峰值速度 | ω=11.31 (State 3) 峰值速度 | 视觉效果 |
|---|---|---|---|
| 5 | 3.47A | 5.05A | 严重过冲，波峰处粒子仍在高速外冲 → 卡顿 |
| 20 | 2.49A | 3.38A | 改善但仍有可见过冲 |
| 40 | 1.37A | 2.06A | 轻微过冲，几乎不可见 |
| 60 | 0.93A | 1.57A | 极小过冲 |
| ∞ | 0 | 0 | 零过冲（直接赋值） |

k=5 时粒子在目标峰值处仍有 3.47A 的外冲速度，目标已停止但粒子仍在冲 → 过冲 → 反向时突跳 → **卡顿感**。中心区域目标速度最大，lerp 跟踪误差相对小 → 看起来平滑。这就是"中心平滑、外围卡顿"的根因。

### 经验

**lerp 追踪周期性目标时，k 必须远大于目标频率 ω**。`k > 5ω` 时振幅衰减 <2%，相位滞后 <11°。**优先用高 k lerp 而非直接赋值**——前者保留 fade plane 拖尾机制的兼容性，后者破坏渲染管线。直接赋值看似"零延迟"但代价是闪烁，得不偿失。

## 四、直接赋值 + Fade Plane = 残影闪烁

### 问题

粒子位置直接赋值后，画面出现大量残影闪烁。

### 根因

Fade plane 的工作原理：每帧先覆盖一层半透明黑（opacity 0.15）让旧帧变暗，再画新帧。lerp 模式下粒子移动缓慢，新旧帧位置接近 → 拖尾效果。直接赋值时粒子每帧跳到新位置，旧帧粒子还在 85% 亮度 → 残影闪烁。

### 修复

直接赋值模式下，fade plane opacity 改为 1.0（完全不透明黑 = 等效清屏）：

```js
const fadeMat = this.fadePlane.material as THREE.MeshBasicMaterial
fadeMat.opacity = (s === UIState.Listening || s === UIState.ActionSpeaking) ? 1.0 : 0.15
```

### 经验

**Fade plane 的拖尾效果依赖粒子帧间位移量小**。直接赋值模式下位移量大，必须将 opacity 提到 1.0 或改用 `renderer.clear()` 清屏。但注意 `renderer.clear()` 配合 `alpha: true` 会清成透明而非黑色，造成白闪——此时只能用 fade plane opacity=1.0 的方案。

## 五、AudioContext 采样率陷阱

### 问题

STT 识别完全失败，无一句正确。

### 根因

将 `AudioContext({ sampleRate: 24000 })` 改为 `new AudioContext()` 后，浏览器默认采样率 48000Hz。`ScriptProcessorNode` 的输出跟随 AudioContext 采样率，代码将 48kHz 的 float32 数据转 int16 后仍声称 `sample_rate: 16000` 发给后端。后端按 16kHz 解析 → 语速慢 3 倍 + 音调严重失真 → STT 无法识别。

### 修复

在 `onaudioprocess` 回调中先降采样再转 int16：

```js
private downsampleFloat32(samples: Float32Array, fromRate: number, toRate: number): Float32Array {
  if (fromRate === toRate) return samples
  const ratio = fromRate / toRate
  const newLen = Math.round(samples.length / ratio)
  const result = new Float32Array(newLen)
  for (let i = 0; i < newLen; i++) {
    const srcIdx = i * ratio
    const lo = Math.floor(srcIdx)
    const hi = Math.min(lo + 1, samples.length - 1)
    const frac = srcIdx - lo
    result[i] = samples[lo] * (1 - frac) + samples[hi] * frac
  }
  return result
}
```

### 经验

**永远不要假设 AudioContext 的采样率**。`new AudioContext()` 的默认采样率由浏览器/OS 决定（48000Hz / 44100Hz / 96000Hz 均可能）。发送音频数据前必须显式降采样到目标采样率。

## 六、Tauri v1 不支持 backgroundColor

### 问题

`tauri.conf.json` 中设置 `"backgroundColor": "#000000"` 后前端无法启动，报错 `Additional properties are not allowed`。

### 根因

`backgroundColor` 是 Tauri v2 的新特性，v1 不支持。

### 修复

从 `tauri.conf.json` 移除 `backgroundColor`，防闪烁改由 HTML/CSS 层面处理：

```css
/* index.html */
html, body, #app { background: #000; }

/* JarvisOrb.vue */
.orb-container { background: #000 !important; }
```

### 经验

**Tauri v1 与 v2 配置项不完全兼容**。添加配置前先查官方文档确认版本支持情况。防白屏闪烁应从最外层（HTML）到最内层（canvas）逐层设黑色背景。

## 七、前端首帧白闪的完整防护链

### 问题

前端首次加载时出现白闪。

### 根因

从 Tauri 窗口创建到 Three.js 首帧渲染之间存在 ~100-300ms 时间差，全链路任何一层白色背景都会透出。

### 修复（四层防护）

| 层级 | 文件 | 修复 |
|---|---|---|
| HTML | index.html | `html, body, #app { background: #000 }` |
| Vue | App.vue | `background: #000` |
| 容器 | JarvisOrb.vue | `background: #000 !important` |
| Canvas | ParticleEngine.ts | `alpha: true` + CSS 黑底 + 首帧只清屏 |

### 经验

**防白闪必须全链路覆盖**。任何一层遗漏都会在特定时序下透出白光。关键是：canvas 设 `alpha: true` 让背景透出，而背景一定是黑色。

## 八、STT/TTS 线程安全：双重加载导致 OOM Kill

### 问题

后端启动时 STT 模型被加载两次，显存占用翻倍，触发 OOM Kill。

### 根因

WebSocket 连接建立时启动后台线程加载 STT，加载过程中 `_loaded` 仍为 `False`，后续 `ensure_stt_loaded()` 再次进入 `load()`，导致模型被加载两次。

### 修复

添加 `_loading` 标志 + `threading.Lock()` 双重检查：

```python
class STTEngine:
    def __init__(self):
        self._loading = False
        self._load_lock = threading.Lock()

    def load(self):
        with self._load_lock:
            if self._loaded or self._loading:
                return
            self._loading = True
        try:
            # ... 加载模型 ...
            self._loaded = True
        finally:
            self._loading = False
```

### 经验

**Python 多线程下懒加载必须加锁**。`if not loaded: load()` 的 check-then-act 不是原子操作，两个线程可以同时通过 check。用 `_loading` 标志防止并发进入加载函数。

## 九、VRAM 降级不应挂起独立进程的 TTS

### 问题

WSL 中 VRAM 占用 91% 触发降级，挂起了 TTS GPU 推理，但 GPT-SoVITS API 在 Windows 端独立运行，不受 WSL 显存控制。

### 修复

```python
def degrade(self):
    if self.backend == "gpt_sovits":
        logger.info("[TTS] GPT-SoVITS API 后端，跳过本地降级（API 是独立进程）")
        return
    self._degraded = True
```

### 经验

**资源降级策略必须区分本地推理与远程 API**。本地推理才占用当前进程的 GPU 资源，远程 API 的 GPU 使用不受当前进程控制。

## 十、WSL2 跨环境网络探测

### 问题

WSL 中无法通过 `127.0.0.1` 访问 Windows 端的 GPT-SoVITS API。

### 修复

多候选 IP + TCP 端口探测：

```python
@staticmethod
def _get_candidate_ips():
    candidates = []
    # 1. ip route 默认网关
    try:
        result = subprocess.run(["ip", "route"], capture_output=True, text=True, timeout=5)
        for line in result.stdout.splitlines():
            if "default" in line:
                candidates.append(line.split()[2])
    except Exception:
        pass
    # 2. resolv.conf nameserver
    try:
        with open("/etc/resolv.conf") as f:
            for line in f:
                if line.startswith("nameserver"):
                    candidates.append(line.split()[1])
    except Exception:
        pass
    # 3. localhost
    candidates.append("127.0.0.1")
    return list(dict.fromkeys(candidates))  # 去重保序
```

### 经验

**WSL2 网络拓扑不固定**。不同 WSL 版本、不同 Windows 版本下，Windows 主机 IP 可能是 `172.x.x.1`（网关）、`10.255.255.254`（DNS）或 `127.0.0.1`（镜像模式）。必须多候选探测，不能硬编码。同时 Windows 端 API 必须绑定 `0.0.0.0` 而非 `127.0.0.1`。

## 十一、GPT-SoVITS numba JIT 编译卡死

### 问题

GPT-SoVITS API 启动时卡在 `librosa` 导入阶段，CPU 时间持续增长但内存不增长。

### 根因

`librosa` 依赖 `numba` JIT 编译，在当前环境下编译卡住。设置 `NUMBA_DISABLE_JIT=1` 后 `librosa` 导入从 4 小时+ 降至 2.3 秒，但 `resampy.interpn` 的 `@guvectorize(nopython=True)` 在 JIT 禁用后仍尝试 nopython 编译，抛出类型推断错误（HTTP 500）。

### 修复

1. `start_jarvis_tts.bat` 设置 `NUMBA_DISABLE_JIT=1`
2. Patch `resampy/interpn.py`：当 JIT 禁用时回退到纯 Python 实现

### 经验

**numba JIT 禁用不等于所有 numba 装饰器自动失效**。`@guvectorize(nopython=True)` 等装饰器仍会尝试编译，必须手动 patch 添加回退逻辑。JIT 禁用不影响 GPU 量化（float16/int8 是 PyTorch 层面的，与 numba 无关）。

## 十二、TTS 流式合成降低内存与延迟

### 问题

TTS 合成整段音频后一次性发送，长文本内存峰值 2.3MB，首字延迟 ~6 秒。

### 修复

GPT-SoVITS API 支持 `streaming_mode`，逐 chunk 读取并推送：

```python
def synthesize_stream(self, text: str):
    """流式合成，逐 8KB chunk 产出 PCM"""
    payload = {**self.GPT_SOVITS_PARAMS, "text": text, "streaming_mode": 3}
    req = urllib.request.Request(url, data=json.dumps(payload).encode())
    resp = urllib.request.urlopen(req, timeout=30)
    # 读取到 WAV header 结束后，逐 chunk yield 纯 PCM
    ...
```

### 经验

**流式合成对语音助手至关重要**。用户等待首字响应的容忍度 ~1 秒，整段合成模式对长文本不可接受。`streaming_mode=3` 是 GPT-SoVITS API v2 的最快响应模式。

## 十三、前端 AudioPipeline 播放卡顿与爆音

### 问题

TTS 音频播放卡顿，块间有爆音。

### 根因

1. `AudioContext` 采样率硬编码 24000Hz，与 GPT-SoVITS 输出 32000Hz 不匹配
2. 音频块间存在 ~16ms 播放间隙
3. 块边界处阶跃信号导致爆音

### 修复

```js
// 1. 使用浏览器默认采样率，createBuffer 传入实际采样率
const ctx = new AudioContext()  // 默认 48000Hz
const buffer = ctx.createBuffer(1, pcm.length, 32000)  // 后端实际采样率

// 2. 精确时间调度，无间隙衔接
source.start(nextPlayTime)
nextPlayTime += buffer.duration

// 3. 64 样本 crossfade 消除块边界爆音
const fadeLen = 64
for (let i = 0; i < fadeLen; i++) {
  const t = i / fadeLen
  prevEnd[i] *= (1 - t)   // 前块尾部淡出
  currStart[i] *= t        // 当前块头部淡入
}
```

### 经验

**Web Audio API 的 `AudioBufferSourceNode` 必须用精确时间调度**。`source.start()` 传入绝对时间而非相对延迟，配合 `nextPlayTime` 追踪确保块间零间隙。crossfade 是消除爆音的标准手段。

## 十四、完整流程.txt 作为唯一设计基准

### 问题

### 教训

| 原始设计（完整流程.txt） | 我做的"优化" | 结果 |

## 十五、音量波动 + fade 衰减速率：残影的双重来源

### 问题

速度预测方案（方案 C）解决了波峰卡顿，但用户仍报告"残影严重"。

### 根因

残影有两层独立来源：

**来源 1：vol 无平滑导致 lerp 目标突变**

```js
// ❌ setVolume 直接赋值
setVolume(v: number, f: number = 0): void {
  this.volumeRMS = v
}
```

麦克风 RMS 每帧高频波动（环境噪声、语音爆发、AGC 抖动）。`maxExpansion = vol * 40` 随之突变 → lerp 目标 `dynamicRadius` 突变 → 粒子位置抖动。突变的瞬间位移远大于稳态位移，旧位置残影在 fade 0.15 下持续 ~10 帧 → 残影严重。

**来源 2：fade=0.15 衰减太慢**

fade=0.15 → 旧帧保留 85% → 残影衰减时间常数 `τ = -1/ln(0.85) ≈ 6.15 帧`。

| 帧数 | 残影亮度 |
|---|---|
| 5 | 44.4% |
| 10 | 19.7% |
| 15 | 8.7% |
| 20 | 3.9% |

State 1/3 波峰处 lerp 粒子每帧位移 ~3-5px（vol=0.5-0.8），接近粒子大小 ~4.3px。在 fade 0.15 下，5 帧前的旧位置距当前位置 15-25px，亮度 44% → 可见离散残影点。

### 修复

**双重修复**：

```js
// 1. vol lerp 平滑（k=10，时间常数 0.1s）
setVolume(v: number, f: number = 0): void {
  this.targetVolumeRMS = v  // 不直接赋值
}

// animate 中
const volLerpK = 1 - Math.exp(-10.0 * dt)
this.volumeRMS += (this.targetVolumeRMS - this.volumeRMS) * volLerpK

// 2. fade opacity 0.15 → 0.3
const fadeMaterial = new THREE.MeshBasicMaterial({
  color: 0x000000,
  transparent: true,
  opacity: 0.3,  // 残影衰减时间常数 2.8 帧
  depthTest: false,
})
```

fade=0.3 的残影衰减：

| 帧数 | 残影亮度 |
|---|---|
| 5 | 16.8% |
| 8 | 5.8% |
| 10 | 2.8% |

5 帧后残影亮度从 44% 降到 16.8%，8 帧后基本不可见。配合 vol 平滑消除突变位移，残影消除。

### fade 值选择依据

fade 值不能太高（=1.0 等于清屏，无流光效果），也不能太低（=0.15 残影严重）。

| fade | 衰减常数 | 5帧残影 | 拖尾效果 | 适用场景 |
|---|---|---|---|---|
| 0.15 | 6.15帧 | 44% | 长拖尾 | 粒子速度慢（State 0） |
| 0.3 | 2.8帧 | 16.8% | 中拖尾 | 粒子速度中-快（State 1/3） |
| 0.5 | 1.44帧 | 3.1% | 短拖尾 | 粒子速度极快 |
| 1.0 | 0帧 | 0% | 无拖尾 | 清屏 |

当前统一用 0.3。State 0 粒子慢，拖尾本就短，0.3 下仍有可见流光；State 1/3 粒子快，0.3 下残影 8 帧内消失。

### 经验

**fade plane 的 opacity 必须与粒子运动速度匹配**。原始设计 fade=0.15 假设粒子位移小（<<粒子尺寸），但 State 1/3 的 vol 驱动位移在 vol 较大时接近或超过粒子尺寸，0.15 的衰减速率不足以消除残影。

**麦克风 RMS 必须平滑后再驱动视觉**。音频帧级 RMS 高频波动大，直接驱动 `maxExpansion` 会导致粒子目标位置抖动。lerp 平滑（k=10，时间常数 0.1s）符合人耳对音量变化的感知分辨率（~100ms），既消除抖动又不损失视觉响应性。

**残影问题诊断顺序**：先检查位移源（vol 是否平滑、k 是否过高、τ 是否过大），再检查 fade 衰减速率。速度预测（方案 C）只解决相位滞后，不解决位移突变和 fade 衰减速率。
