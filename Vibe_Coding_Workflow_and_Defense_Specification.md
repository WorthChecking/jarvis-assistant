# Vibe Coding 开发工作流与防御规范说明书

## 【工作流基线前提】

* **核心AI编码工具选型**：使用 Trae IDE。
* **版本控制与回滚粒度**：强制要求进行“原子化 Commit”。
* **本地环境安全隔离**：计划配置沙盒（Windows Sandbox）进行初期的调试与测试。

---

## 版块一：环境准备与 Trae 上下文挂载策略

### 1. 项目目录初始化 (Monorepo结构)
* **规范**：在开发环境（及测试沙盒）中创建一个统一的根目录（例如命名为 `Jarvis_Assistant`），并在其下严格建立两个独立的子工作区：
    * `/frontend`：用于存放 Tauri + Vue 3 的前端代码。
    * `/backend`：用于存放 Python 的中控服务、AI 模型及 ChromaDB 数据库代码。
* **目的**：避免 Trae 在全局扫描时混淆前端的 TypeScript/Node 逻辑与后端的 Python 逻辑，从物理层面上为 AI 划定明确的代码边界。

### 2. 核心架构文档强制注入
* **操作**：将完整的 `PRD_and_Architecture_v1.0.md` 文件放置在项目根目录中。
* **约束机制**：在开始让 Trae 生成代码之前，必须在 Trae 的对话框或系统提示词中声明一条绝对指令：“**在进行任何需求分析、代码生成或重构前，必须优先读取 `@PRD_and_Architecture_v1.0.md` 文件，所有技术选型和功能边界必须以该文档为最高准则。**”

### 3. 配置项目级 AI 规则约束 (AI Rules)
为了防止 AI 在长对话中发生“灾难性遗忘”或产生危险幻觉，必须在 Trae 中设置项目级的系统提示词（System Prompt），强制强调三大红线：
* **资源红线**：“本机显存上限为 RTX 3060 6GB。在生成 Faster-Whisper 和 TTS 的加载代码时，必须强制指定量化参数（int8/float16）和显存限制策略。”
* **安全红线**：“严禁生成任何可能修改、删除系统核心文件（如 `C:\Windows`）或格式化磁盘的 Python 脚本。系统操作必须严格遵照 PRD 中的白名单执行。”
* **渲染红线**：“在处理前端波形 UI 时，仅允许使用 HTML5 Canvas API 进行绘制，严禁使用 DOM 节点或纯 CSS 动画进行高频渲染。”

## 版块二：前后端解耦的 Vibe Coding 实施路径

### 1. 阶段一：通信协议契约生成 (API-First)
* **操作规范**：在开始编写任何业务逻辑前，首先在根目录创建 `communication_protocol.md` 或相关的 JSON Schema 文件。必须明确 Python 端发送音频响度数据（用于驱动波形）和状态切换信号的具体数据结构，并在前端生成对应的 TypeScript Interface。
* **卡点**：在两端数据结构完全对齐且测试通过前，严禁进入下一阶段。

### 2. 阶段二：后端中控层独立生成与验证 (Python Backend)
* **模型挂载与降级测试**：引导 Trae 依次编写 SenseVoiceSmall、TTS 与 ChromaDB 的初始化代码。在生成推理代码时，紧盯量化参数与显存释放逻辑，如有遗漏立即打断重构。
* **白名单安全沙盒**：生成系统操作前，要求先生成包含所有黑名单的防御性代码，再生成实际的执行逻辑。
* **脱离前端测试**：使用 Python 写一个简易的 CLI 测试脚本模拟前端请求，跑通整个流水线。

### 3. 阶段三：前端表现层独立生成 (Tauri + Vue 3)
* **窗口与透明穿透**：引导 Trae 生成 Tauri 配置及 Rust 底层代码，完成无边框、透明及鼠标穿透配置。
* **Canvas 波形渲染器**：明确指令使用 HTML5 Canvas API 和 requestAnimationFrame 实现，严禁使用 Vue 响应式变量高频绑定 DOM 节点或 CSS 动画。
* **状态机流转**：严格按照四种 State 编写 Vue 端的有限状态机（FSM）逻辑。使用 Mock 数据测试波形。

### 4. 阶段四：端到端联调与性能压测
* **操作规范**：生成联调启动脚本。
* **资源监控**：必须开启系统的任务管理器与 `nvidia-smi` 面板，观察资源释放与显存占用。若出现严重掉帧或 OOM，将日志喂给 Trae 进行优化。

## 版块三：本地沙盒调试与系统权限防御规范

1. 物理隔离：独立虚拟机 (VMware/VirtualBox) 的配置与挂载
规范：严禁在宿主机（你的真实电脑）的物理环境中直接运行并测试涉及文件系统检索、应用启停控制的 Python 脚本。

实施路径：

采用 VMware Workstation Player 或 Oracle VirtualBox 部署一个纯净的 Windows 10/11 虚拟机（Guest OS）作为专属测试机。

启用虚拟机的“共享文件夹 (Shared Folders)”功能，将宿主机的开发目录（Jarvis_Assistant）映射到虚拟机内部网络盘符。

强制准则：在宿主机的 Trae IDE 中仅进行代码生成与修改。所有针对 Python 后端涉及系统调用的“运行与测试”动作，必须在虚拟机内部的终端中手动执行。建议在虚拟机部署完毕后立刻创建一个“初始纯净快照”，一旦 AI 生成的恶意或错误脚本导致虚拟机系统崩溃，立即回滚快照。

2. 代码级防御：AOP（面向切面）拦截器的首优开发
规范：在让 Trae 生成任何涉及大模型 (LLM) 功能调用的代码前，必须优先让它编写底层防御代码。使用 Python 的 unittest.mock 或直接重写标准库的方式，构建全局安全拦截器。

强制拦截名单：强制拦截并禁用 os.remove、shutil.rmtree、os.rename 以及 subprocess.Popen 中任何带有危险标志的命令行参数。即使 LLM 输出并尝试执行这些函数，也只能打印一条 Warning 日志而不产生实际系统级影响。

3. LLM API Key 与环境变量的硬隔离
防范规范：建立严格的 .env 机制，第一时间将 .env 加入 .gitignore。严禁在日志、输出或返回前端的错误信息中打印环境变量。必须使用代码动态获取（如 os.getenv），严禁硬编码。

4. 读写权限的最低特权原则 (Least Privilege)
机制：实现“安全沙盒目录检查”逻辑。传入绝对路径前，先校验是否位于系统盘核心敏感目录（如 C:\Windows 等）。若命中黑名单，直接抛出 PermissionError。

## 版块四：原子化 Git 提交流程与回滚机制

### 1. 适用独立开发的单机分支模型
* **规范**：采用极简分支策略。`main` 分支永远处于可运行状态，严禁在此分支下直接生成代码。必须切出 `feat/` 或 `fix/` 分支，沙盒测试通过后方可合并。

### 2. 强制原子化提交 (Atomic Commits)
* **规范**：绝对禁止粗粒度提交。每完成一个极小且独立的可运行节点，未报错即刻 Commit。
* **信息规范**：强制生成清晰的 Conventional Commits 格式信息。

### 3. AI 幻觉熔断与硬回滚策略 (Hard Reset)
* **熔断条件**：AI 连续两轮无法修复 Bug 或试图大面积重构掩盖报错。
* **执行操作**：立即停止对话，执行 `git reset --hard HEAD` 或 `git checkout`，重新明确目标并从正确基线生成。

### 4. `.gitignore` 的绝对红线
* **规范**：严格排除本地向量数据库文件（`/.chroma`）、AI 模型权重文件、包含 API Key 的 `.env` 文件、Python 虚拟环境与 Node 依赖。
